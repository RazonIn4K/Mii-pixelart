# Security policy

Thanks for taking the time to look at this project's security. Because Tomodachi runs a recovery hub for the Tomodachishare credential leak, the project takes responsible disclosure seriously even though it's a solo-maintained fan tool.

## Reporting a vulnerability

**Please email [security@tomodachi.pw](mailto:security@tomodachi.pw)** with a clear description of the issue and steps to reproduce. PGP is not currently required, but if you have a key you'd like to use, mention it in your first message and we will exchange keys before the technical details.

**Please do NOT:**

- Open a public GitHub issue for a vulnerability that hasn't been patched
- Use the AI assistant, paywall, or other production endpoints to demonstrate the issue in ways that exfiltrate or modify other people's data
- Run automated scanners that materially degrade service for other users (basic Nuclei / nikto / Burp passive scans against your own requests are fine)

We aim to acknowledge new reports within **72 hours** and provide a fuller response (severity assessment, planned remediation timeline, or request for more information) within **7 days**.

## Scope

In scope:

- `tomodachi.pw` and any subdomains (including `www.tomodachi.pw`)
- `tomodachi.brave` (the IPFS mirror)
- Cloudflare Pages Functions at `/api/*` (AI proxy, Stripe checkout/session/webhook, model list)
- This GitHub repository's CI / build pipeline configuration

Out of scope:

- Any Nintendo, Tomodachi Life, or 3DS hardware / software (this is an unofficial fan tool)
- Third-party services (OpenRouter, Stripe, Cloudflare, HIBP, Doppler) — please report directly to those vendors
- Self-XSS that requires the user to paste attacker-controlled content into their own browser DevTools console
- Findings that depend solely on configuration the user controls (their own browser, their own extensions, their own AdBlock rules)
- Reports generated by automated tools without manual verification or a working proof-of-concept

## What we appreciate

- A working proof-of-concept (a curl command, a request snippet, or a recorded session)
- A suggested CVSS v3.1 vector and severity rating (we may adjust based on real-world impact)
- A few days to remediate before public disclosure (we will coordinate timelines)

## What we don't currently offer

Tomodachi is solo-funded by Stripe tips and consult bookings, so we cannot offer monetary bug bounties at this time. We will gladly:

- Credit you publicly in the [release notes](./CHANGELOG.md) and a `SECURITY-THANKS.md` once we have a first qualifying report
- Provide a public acknowledgement on the [`/about`](https://tomodachi.pw/about) page
- Write a recommendation on LinkedIn or a similar professional venue if you'd like one

## What's already hardened

For context on prior security work, see the audit-trail documents under [`docs/`](./docs/):

- Content Security Policy is locked down per Google CSP Evaluator with Trusted Types in report-only mode (see [`client/public/_headers`](./client/public/_headers))
- Stripe checkout uses an origin allowlist (`assertStripeRedirect` in [`client/src/lib/stripeUrl.ts`](./client/src/lib/stripeUrl.ts)) to block open-redirect attacks
- Password breach checks use HIBP k-anonymity (only the first 5 chars of SHA-1 ever leave the browser)
- Stripe webhook signatures are verified with HMAC-SHA256 at the edge before any state changes
- AdSense and AI calls only fire after explicit user consent through the cookie banner
- `frame-ancestors 'none'` + `X-Frame-Options: DENY` prevent clickjacking
- Strict-Transport-Security: 1-year max-age, includeSubDomains, preload-eligible
